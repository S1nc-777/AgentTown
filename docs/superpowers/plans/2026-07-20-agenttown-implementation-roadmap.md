# AgentTown Implementation Roadmap

> 本路线图把已确认规格拆成七个可独立验收的子项目。每个子项目开始前都要有单独的详细实现计划；只有阶段 0 已在本次规划中展开到可执行步骤。

**总目标：** 在 Windows 上交付一个本地优先、后台持续运行、可调度 Codex CLI 与 Claude Code CLI 的多 Agent“赛博公司”。

**拆分方法：** 使用 WBS 保证阶段边界互斥且覆盖首版范围，使用 MVP/BML 先验证最危险的 PTY、会话恢复与异构 Agent 接入假设，再投入完整控制层和可视化界面。

## P0：Windows PTY 与技术选型

**可交付成果：** 可重复运行的探针套件、Codex/Claude 能力报告、Electron 与 Tauri 对比数据、正式 ADR。

**验收门槛：** 两种 Agent 均完成启动、流式输出、会话 ID 捕获、续聊、中断和退出；至少三个会话并行运行；技术路线由数据而非偏好决定。

详细计划：`docs/superpowers/plans/2026-07-20-agenttown-windows-agent-feasibility.md`

## P1：确定性后台内核

**可交付成果：** 独立后台进程、版本化本地 IPC、SQLite 事实库、事件日志、公司/员工/会话生命周期、脚本化假 Agent。

**验收门槛：** CLI 可创建一家公司并启动四个模拟员工；UI 未运行时内核继续工作；重启后从 SQLite 恢复事实状态。

## P2：Codex 与 Claude Code 适配器

**可交付成果：** 统一 `AgentAdapter` 接口、Codex 与 Claude Code 正式适配器、能力声明、原始日志、结构化事件和会话恢复。

**验收门槛：** 两个适配器通过同一契约测试；无法原生恢复时明确产生重建事件，不能伪装成原会话。

## P3：任务、通信与确定性策略

**可交付成果：** 任务状态机、领导派工、员工消息路由、权限审批、预算与循环限制、用户接管锁。

**验收门槛：** 领导只能选择固定名单；越权、超预算、重复消息和接管中的自动投递都被确定性拒绝并记录原因。

## P4：工作区、审核、记忆与恢复

**可交付成果：** Git Worktree 管理、变更集集成、测试/审核返工、公共记忆、上下文摘要、崩溃恢复清单。

**验收门槛：** 开发员工不能自审；两次审核失败升级用户；主工作区与员工 Worktree 的更改不会静默覆盖。

## P5：桌面端、CLI、托盘与模板

**可交付成果：** 赛博办公室、任务看板、时间线、员工抽屉、嵌入终端、系统托盘、CLI、标准四人小队和极简模板。

**验收门槛：** 初次配置能在十几分钟完成；用户默认只与领导沟通，但可接管任意员工；关闭窗口不停止公司。

## P6：真实项目验收与开源发布

**可交付成果：** Windows 安装包、升级/卸载验证、真实小项目演示、双语 README、AGPL-3.0 LICENSE、贡献指南、安全说明和故障排查。

**验收门槛：** 使用真实 Codex 与 Claude Code 跑通“需求确认—拆分—编码—测试—审核—返工—交付”；全新 Windows 用户按 README 可复现演示。

## 跨阶段规则

- 每阶段都先写失败测试，再实现最小能力，最后跑完整验证。
- 每个阶段以可运行产物结束，不提交只有接口没有行为的空壳。
- 原始终端日志、事实状态和用户可见状态必须能互相追溯。
- 新能力不能绕过固定员工名单、权限、审批、预算或接管锁。
- 普通 CI 只使用脚本化假 Agent；真实模型测试由维护者手动触发。
- 阶段验收失败时停在当前阶段修正，不把未知风险转移给下一阶段。
